## 1. Как использовать этот план

### 1.1 Роли

| Роль | Ответственность |
|------|-----------------|
| **Пользователь** | Предоставляет реальные данные (TG API ID/Hash, QR-код сканирование, API-ключи), валидирует результаты |
| **Агент** | Выполняет API-запросы, проверяет ответы, логирует результаты |

### 1.2 Формат сценария

Каждый сценарий содержит:
- **Цель** — что проверяем
- **Предусловия** — что должно быть готово
- **Шаги** — кто что делает (Пользователь / Агент)
- **Ожидаемый результат** — что должно произойти
- **Критерий приёмки** — как понять что тест пройден

### 1.3 Взаимодействие

```
Пользователь: предоставляет данные → Агент: выполняет запрос → Пользователь: проверяет результат
```

**Пример диалога:**
```
Агент: "Отсканируйте QR-код в Telegram"
Пользователь: [сканирует] "Готово"
Агент: "Проверяю статус... Авторизация успешна ✅"
```

## 2. Подготовка окружения

### 2.1 Проверка перед началом

Выполните эти команды и убедитесь что всё работает:

```bash
# 1. Активация venv
venv\Scripts\activate

# 2. Проверка PostgreSQL
docker ps | findstr postgres
# Должен быть запущен контейнер postgres

# 3. Проверка Ollama (если используется)
curl http://localhost:11434/api/tags
# Должен вернуть список моделей

# 4. Запуск приложения
uvicorn main:app --reload --host 0.0.0.0 --port 8000
# Должен запуститься без ошибок

# 5. Проверка health
curl http://localhost:8000/health
# Должен вернуть {"status": "healthy", ...}
```

### 2.2 Swagger UI

Откройте http://localhost:8000/docs — здесь можно查看所有 эндпоинты и тестировать вручную.

### 2.3 Подготовка тестовых данных

Пользователь должен подготовить:

| Данные | Где взять | Формат |
|--------|-----------|--------|
| TG_API_ID | https://my.telegram.org/apps | Число (например `12345678`) |
| TG_API_HASH | https://my.telegram.org/apps | Строка (32 символа) |
| Реальный чат ID | Из Telegram или через API sync | `-100XXXXXXXXXX` |
| LLM API Key | OpenRouter/Gemini/OpenAI | `sk-...` или `AIzaSy...` |
| Webhook URL (опционально) | webhook.site или свой сервер | `https://...` |

## 3. Предварительные требования

### 3.1 Минимальная конфигурация .env

```ini
# === Telegram ===
TG_API_ID=<ваш_api_id>
TG_API_HASH=<ваш_api_hash>

# === Database ===
DB_HOST=localhost
DB_PORT=5432
DB_NAME=tg_db
DB_USER=postgres
DB_PASSWORD=<ваш_пароль>

# === LLM (хотя бы один) ===
LLM_ACTIVE_PROVIDER=gemini
GEMINI_API_KEY=<ваш_gemini_key>
GEMINI_MODEL=gemini-2.5-flash

# === Embedding (хотя бы один) ===
OLLAMA_EMBEDDING_PROVIDER=ollama
OLLAMA_EMBEDDING_URL=http://localhost:11434
OLLAMA_EMBEDDING_MODEL=nomic-embed-text
```

### 3.2 Чек-лист готовности

- [ ] PostgreSQL запущен и доступен
- [ ] Ollama запущен (если используется для embedding)
- [ ] Приложение запущено на port 8000
- [ ] `/health` возвращает `status: "healthy"`
- [ ] TG_API_ID и TG_API_HASH предоставлены
- [ ] LLM API key предоставлен
- [ ] Есть доступ к реальному Telegram чату (группа/канал)

## 4. Сценарии тестирования

### Сценарий 1: Авторизация через QR-код

**Цель:** Проверить полный цикл QR-авторизации в Telegram.

**Предусловия:**
- Приложение запущено
- TG_API_ID и TG_API_HASH настроены в .env
- Пользователь имеет Telegram на телефоне

---

**Шаг 1: Создание QR-сессии**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/settings/telegram/qr-code` |
| Агент | Проверяет ответ: `session_id`, наличие QR-кода |
| Пользователь | Видит QR-код в консоли/ответе |

**Ожидаемый ответ:**
```json
{
  "success": true,
  "session_id": "uuid-...",
  "qr_code": "data:image/png;base64,..."
}
```

---

**Шаг 2: Сканирование QR-кода**

| Кто | Действие |
|-----|----------|
| Пользователь | Открывает Telegram на телефоне |
| Пользователь | Settings → Devices → Link Desktop Device |
| Пользователь | Сканирует QR-код из ответа |
| Пользователь | Подтверждает авторизацию |
| Пользователь | Сообщает агенту: "Готово" |

---

**Шаг 3: Проверка статуса**

| Кто | Действие |
|-----|----------|
| Агент | `GET /api/v1/settings/telegram/qr-status/{session_id}` |
| Агент | Проверяет `status: "authorized"` |
| Агент | Проверяет наличие `phone_number` |

**Ожидаемый ответ:**
```json
{
  "session_id": "uuid-...",
  "status": "authorized",
  "phone_number": "+7XXXXXXXXXX"
}
```

---

**Шаг 4: Проверка сессии**

| Кто | Действие |
|-----|----------|
| Агент | `GET /api/v1/settings/telegram/check` |
| Агент | Проверяет `connected: true` |
| Агент | `GET /api/v1/settings/telegram/auth-status` |
| Агент | Проверяет `authorized: true` |

**Ожидаемый ответ (check):**
```json
{
  "connected": true,
  "session_name": "default"
}
```

**Ожидаемый ответ (auth-status):**
```json
{
  "authorized": true,
  "phone": "+7XXXXXXXXXX",
  "user_id": 123456789
}
```

---

**Критерий приёмки:**
- [ ] QR-код сгенерирован
- [ ] Сканирование прошло успешно
- [ ] Статус сменился на `authorized`
- [ ] Сессия подключена к Telegram
- [ ] Настройки Telegram сохранены в БД

**Время выполнения:** ~3-5 минут

### Сценарий 2: Настройка LLM и Embedding провайдеров

**Цель:** Проверить настройку, активацию и проверку здоровья провайдеров.

**Предусловия:**
- Приложение запущено
- Есть API-ключи хотя бы для одного LLM-провайдера

---

**Шаг 1: Просмотр текущих настроек**

| Кто | Действие |
|-----|----------|
| Агент | `GET /api/v1/settings/overview` |
| Агент | Показывает пользователю текущие настройки LLM/Embedding |
| Пользователь | Подтверждает какие провайдеры нужно настроить |

---

**Шаг 2: Настройка LLM-провайдера (Gemini)**

| Кто | Действие |
|-----|----------|
| Пользователь | Предоставляет GEMINI_API_KEY |
| Агент | `GET /api/v1/settings/llm/gemini` |
| Агент | Показывает текущие настройки |
| Агент | `PUT /api/v1/settings/llm/gemini` с `{ "api_key": "...", "model": "gemini-2.5-flash", "is_enabled": true }` |
| Агент | Проверяет `success: true` |

---

**Шаг 3: Активация и проверка LLM**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/settings/llm/gemini/activate` |
| Агент | Проверяет `is_active: true` |
| Агент | `POST /api/v1/settings/llm/gemini/check` |
| Агент | Проверяет `healthy: true` |

**Ожидаемый ответ (check):**
```json
{
  "healthy": true,
  "provider": "gemini",
  "model": "gemini-2.5-flash",
  "response_time_ms": 234
}
```

---

**Шаг 4: Настройка Embedding-провайдера (Ollama)**

| Кто | Действие |
|-----|----------|
| Агент | `GET /api/v1/settings/embedding/ollama` |
| Агент | Показывает текущие настройки |
| Агент | `PUT /api/v1/settings/embedding/ollama` с `{ "model": "nomic-embed-text", "is_enabled": true }` |
| Агент | `POST /api/v1/settings/embedding/ollama/activate` |
| Агент | `POST /api/v1/settings/embedding/ollama/check` |
| Агент | Проверяет `healthy: true` |

---

**Шаг 5: Проверка embedding dimension**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/settings/embedding/ollama/refresh-dimension` |
| Агент | Проверяет что `embedding_dim` определён (например 768) |
| Агент | Показывает пользователю результат |

---

**Критерий приёмки:**
- [ ] LLM-провайдер настроен и активен
- [ ] LLM health check прошёл
- [ ] Embedding-провайдер настроен и активен
- [ ] Embedding dimension определён
- [ ] Embedding health check прошёл

**Время выполнения:** ~5 минут

### Сценарий 3: Управление чатами

**Цель:** Проверить синхронизацию с Telegram, управление мониторингом и настройками чатов.

**Предусловия:**
- Telegram сессия авторизована (Сценарий 1 пройден)

---

**Шаг 1: Синхронизация чатов**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/settings/chats/sync` |
| Агент | Проверяет `success: true` |
| Агент | Показывает список синхронизированных чатов |
| Пользователь | Подтверждает что видит свои чаты |

**Ожидаемый ответ:**
```json
{
  "success": true,
  "synced": 5,
  "new": 3,
  "updated": 2,
  "chats": [
    {"chat_id": -1001234567890, "title": "Моя Группа", "type": "supergroup"},
    ...
  ]
}
```

---

**Шаг 2: Просмотр всех чатов**

| Кто | Действие |
|-----|----------|
| Агент | `GET /api/v1/settings/chats/list` |
| Агент | Показывает пользователю полный список с метаданными |
| Пользователь | Выбирает чат для тестирования |

---

**Шаг 3: Включение мониторинга**

| Кто | Действие |
|-----|----------|
| Пользователь | Указывает `chat_id` для тестирования |
| Агент | `POST /api/v1/settings/chats/{chat_id}/enable` |
| Агент | Проверяет `is_monitored: true` |

---

**Шаг 4: Настройка Summary для чата**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/settings/chats/{chat_id}/summary/enable` |
| Агент | `PUT /api/v1/settings/chats/{chat_id}/summary/period` с `{"minutes": 60}` |
| Агент | Проверяет `period_minutes: 60` |
| Агент | `PUT /api/v1/settings/chats/{chat_id}/summary/prompt` с кастомным промптом |
| Агент | `GET /api/v1/settings/chats/{chat_id}/summary/prompt` — проверяет сохранение |

**Тестовый промпт:**
```
Сделай краткую выжимку обсуждения на русском языке.
Выдели основные темы и принятые решения.
```

---

**Шаг 5: Переключение мониторинга**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/settings/chats/{chat_id}/toggle` |
| Агент | Проверяет `is_monitored: false` |
| Агент | `POST /api/v1/settings/chats/{chat_id}/toggle` |
| Агент | Проверяет `is_monitored: true` |

---

**Критерий приёмки:**
- [ ] Чаты синхронизированы с Telegram
- [ ] Список чатов отображается корректно
- [ ] Мониторинг включается/отключается
- [ ] Summary настройки сохраняются
- [ ] Кастомный промпт сохраняется

**Время выполнения:** ~5 минут

### Сценарий 4: Внешние сообщения

**Цель:** Проверить приём и обработку сообщений из внешних источников.

**Предусловия:**
- Чат создан и мониторинг включен (Сценарий 3 пройден)
- Embedding-провайдер настроен (Сценарий 2 пройден)

---

**Шаг 1: Отправка валидного сообщения**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/messages/ingest` с тестовым сообщением |
| Агент | Проверяет `success: true`, `message_id` |
| Пользователь | Подтверждает что сообщение принято |

**Тестовое сообщение:**
```json
{
  "chat_id": -1001234567890,
  "text": "Привет! Сегодня обсуждали новый план разработки. Нужно завершить API до конца недели.",
  "date": "2026-04-03T12:00:00Z",
  "sender_id": "123456",
  "sender_name": "Иван Петров",
  "message_link": "https://t.me/c/1234567890/100"
}
```

**Ожидаемый ответ:**
```json
{
  "success": true,
  "status": "processed",
  "message_id": "...",
  "chat_id": -1001234567890,
  "filtered": false,
  "pending": false,
  "duplicate": false
}
```

---

**Шаг 2: Отправка серии сообщений**

| Кто | Действие |
|-----|----------|
| Агент | Отправляет 5-10 сообщений с разным содержанием |
| Агент | Проверяет что каждое обработано |
| Пользователь | Подтверждает что все сообщения уникальны |

**Серия тестовых сообщений:**
```json
[
  {"text": "Встреча назначена на 15:00", "sender_name": "Мария"},
  {"text": "Нужно подготовить презентацию", "sender_name": "Алексей"},
  {"text": "Баг в авторизации исправлен", "sender_name": "Дмитрий"},
  {"text": "Деплой запланирован на пятницу", "sender_name": "Иван Петров"},
  {"text": "Тесты проходят успешно", "sender_name": "Мария"}
]
```

---

**Шаг 3: Проверка дубликата**

| Кто | Действие |
|-----|----------|
| Агент | Отправляет то же сообщение повторно |
| Агент | Проверяет `duplicate: true` |

---

**Шаг 4: Проверка ошибки (чат не мониторится)**

| Кто | Действие |
|-----|----------|
| Агент | Отправляет сообщение с несуществующим `chat_id` |
| Агент | Проверяет ошибку `EXT-002` (HTTP 400) |

---

**Шаг 5: Проверка фильтрации**

| Кто | Действие |
|-----|----------|
| Агент | Отправляет сообщение с пустым текстом или только пробелами |
| Агент | Проверяет `filtered: true` или ошибку валидации |

---

**Критерий приёмки:**
- [ ] Валидные сообщения обрабатываются
- [ ] Embedding генерируется
- [ ] Дубликаты обнаруживаются
- [ ] Невалидные запросы отклоняются
- [ ] Ошибки имеют корректные коды (EXT-001, EXT-002)

**Время выполнения:** ~5 минут

### Сценарий 5: Пакетный импорт

**Цель:** Проверить импорт большого количества сообщений из файла/JSON.

**Предусловия:**
- Чат создан и мониторинг включен
- Embedding-провайдер настроен

---

**Шаг 1: Подготовка тестовых данных**

| Кто | Действие |
|-----|----------|
| Пользователь | Подтверждает `chat_id` для импорта |
| Агент | Создаёт JSON с 20-50 тестовыми сообщениями |

**Пример JSON:**
```json
[
  {
    "message_id": 1,
    "text": "Начало обсуждения проекта",
    "sender_id": "100",
    "sender_name": "Иван",
    "date": "2026-04-01T09:00:00Z"
  },
  {
    "message_id": 2,
    "text": "Предлагаю использовать FastAPI",
    "sender_id": "200",
    "sender_name": "Мария",
    "date": "2026-04-01T09:05:00Z"
  }
]
```

---

**Шаг 2: Запуск импорта**

| Кто | Действие |
|-----|----------|
| Агент | `POST /api/v1/messages/import` с `{ "json_data": [...], "chat_id": -1001234567890 }` |
| Агент | Получает `task_id` |
| Агент | Показывает `task_id` пользователю |

**Ожидаемый ответ:**
```json
{
  "success": true,
  "task_id": "import-uuid-...",
  "status": "pending",
  "total_messages": 50
}
```

---

**Шаг 3: Мониторинг прогресса**

| Кто | Действие |
|-----|----------|
| Агент | `GET /api/v1/messages/import/{task_id}/progress` |
| Агент | Повторяет запрос каждые 2 секунды |
| Агент | Показывает прогресс пользователю |
| Пользователь | Наблюдает за прогрессом |

**Ожидаемый ответ:**
```json
{
  "task_id": "import-uuid-...",
  "status": "processing",
  "progress": 45,
  "processed": 23,
  "total": 50,
  "errors": 0
}
```

---

**Шаг 4: Завершение импорта**

| Кто | Действие |
|-----|----------|
| Агент | Ждёт `status: "completed"` |
| Агент | Проверяет `progress: 100` |
| Агент | Показывает итоговую статистику |
| Пользователь | Подтверждает успешное завершение |

**Ожидаемый ответ:**
```json
{
  "task_id": "import-uuid-...",
  "status": "completed",
  "progress": 100,
  "processed": 50,
  "total": 50,
  "errors": 0,
  "duplicates": 2,
  "completed_at": "2026-04-03T12:05:00Z"
}
```

---

**Шаг 5: Проверка сообщений в БД**

| Кто | Действие |
|-----|----------|
| Агент | `GET /api/v1/settings/chats/{chat_id}` |
| Агент | Проверяет что `last_message_id` обновился |
| Агент | `GET /api/v1/settings/reindex/stats` |
| Агент | Проверяет что количество векторов увеличилось |
| Пользователь | Подтверждает что данные импортированы |

---

**Шаг 6: Тест отмены импорта (опционально)**

| Кто | Действие |
|-----|----------|
| Агент | Запускает новый импорт с большим количеством сообщений |
| Агент | Через 1-2 секунды: `DELETE /api/v1/messages/import/{task_id}/cancel` |
| Агент | Проверяет `status: "cancelled"` |
| Пользователь | Подтверждает что импорт остановлен |

---

**Критерий приёмки:**
- [ ] Импорт запущен успешно
- [ ] Прогресс отображается корректно
- [ ] Все сообщения импортированы
