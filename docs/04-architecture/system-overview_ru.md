## Архитектура верхнего уровня

TgBrain следует многоуровневой архитектуре с чётким разделением ответственности:

```
+-------------------+
|    API Layer      |  FastAPI роутеры, модели запросов/ответов
+-------------------+
|  Application      |  Бизнес-сервисы, оркестрация
+-------------------+
|    Domain         |  Объекты-значения, доменные исключения
+-------------------+
|  Infrastructure   |  База данных, внешние сервисы, персистентность
+-------------------+
```

## Фоновые сервисы

Приложение запускает несколько долгоживущих фоновых сервисов в виде asyncio задач:

| Сервис | Описание |
|--------|----------|
| TelegramIngester | Опрос Telegram на наличие новых сообщений |
| ReindexService | Регенерация эмбеддингов при смене модели |
| SummaryTaskService | Управление асинхронной генерацией саммари |
| SummaryEmbeddingsService | Генерация эмбеддингов для саммари |
| SummaryCleanupService | Очистка старых задач саммари |
| ScheduleService | Запуск генерации саммари по расписанию |
| PendingCleanupService | Очистка устаревших отложенных сообщений |
| WebhookService | Доставка саммари через вебхуки |

## Последовательность запуска

1. Загрузка настроек из базы данных (переопределение .env)
2. Инициализация пула соединений с базой данных
3. Создание клиентов embedding и LLM
4. Инициализация репозиториев
5. Запуск фоновых сервисов
6. Проверка Telegram-сессии и запуск ingestion при успешной аутентификации

## Последовательность остановки

1. Остановка ScheduleService
2. Закрытие WebhookService
3. Остановка TelegramIngester
4. Закрытие пула соединений с базой данных

## Поток данных

### Приём сообщений

```
Telegram --> Ingester --> Filter --> Embedder --> Database
```

### RAG-поиск

```
User Query --> Embedder --> Vector Search --> Context Expand --> LLM --> Answer
```

### Генерация саммари

```
API/Schedule --> Message Fetch --> LLM --> Summary Store --> Embedder --> Webhook
```
