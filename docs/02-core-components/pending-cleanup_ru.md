# Pending Cleanup
Языки: [English](pending-cleanup.md) | [Русский](pending-cleanup_ru.md)

## Обзор

Служба очистки отложенных сообщений управляет сообщениями, которые не удалось немедленно обработать во время ingestion. Сообщения помещаются в таблицу `pending_messages`, когда происходит сбой эмбеддинга или другие временные ошибки.

## Как это работает

1. Сообщения, у которых не удалось создать эмбеддинг во время ingestion, сохраняются в `pending_messages`
2. PendingCleanupService периодически сканирует таблицу на наличие устаревших отложенных сообщений
3. Сообщения, превысившие TTL, удаляются
4. Настройки хранятся в таблице `app_settings` с ключами `pending.*`

## Конфигурация

| Ключ настройки | По умолчанию | Описание |
|-------------|---------|-------------|
| `pending.ttl_minutes` | 240 | Время жизни отложенных сообщений (4 часа) |
| `pending.cleanup_interval_minutes` | 60 | Частота запуска очистки (1 час) |

Настройки можно изменять через App Settings API:

```bash
# Обновить TTL
curl -X PUT http://localhost:8000/api/v1/settings/app/pending.ttl_minutes \
  -H "Content-Type: application/json" \
  -d '{"value": 480}'

# Обновить интервал
curl -X PUT http://localhost:8000/api/v1/settings/app/pending.cleanup_interval_minutes \
  -H "Content-Type: application/json" \
  -d '{"value": 30}'
```

## Источники отложенных сообщений

Отложенные сообщения поступают из двух источников:

1. **Telegram ingestion** -- сообщения из отслеживаемых чатов, у которых не удалось создать эмбеддинг
2. **External ingestion** -- сообщения из API, у которых не удалось создать эмбеддинг (сохраняются со статусом pending)

## Поведение очистки

Служба очистки:
- Запускается с заданным интервалом
- Удаляет сообщения старше TTL
- Логирует количество очищенных сообщений
- Не пытается повторную обработку (сообщения удаляются, если они превысили TTL)

## Мониторинг

Количество отложенных сообщений можно наблюдать через эндпоинт системной статистики. Растущее количество отложенных сообщений может указывать на проблемы со службой эмбеддингов.
